## 3. 시스템 출력 정의

### 3.1. 출력 정의 상태 (2026-05-14 기준)

본 항목의 정식 출력 정의는 **B 방향 전환 직후 단계에서는 확정하지 않는다**.
다음 단계의 PC 작업 1~2단계 가능성 확인이 끝난 뒤 확정한다.

확인 범위:
1. pykrx 로 한국 상장 ETF 전체 universe 조회 가능 여부.
2. ETF 별 일간 / 1개월 / 3개월 수익률 계산 가능 여부.
3. 수익률 기준 TOP N 산출 가능 여부.

TOP N 의 N 값은 고정하지 않고 운영/테스트 중 변경 가능하게 둔다.

PC 작업의 핵심 방향 후보 (출력 정의 확정 전 단계):
- 섹터/테마 발굴 후보 산출 (한국 상장 ETF universe 기반).
- AI 투자세션에서 사용자가 직접 해석할 수 있는 정량 재료 제공.
- ML 연결 여부는 위 1~3 가능성 확인 결과에 따라 별도 판단.

### 3.2. 보조 출력 — 기존 Telegram PUSH 배관

기존 3-PUSH 구조는 **메인 출력이 아니라 보조 알림 배관**으로 유지한다.
이 분류는 Step7 시리즈에서 운영화된 배관을 **삭제하지 않고 보존**하기 위함이다.

- 보유 종목 상태 브리핑 (PUSH 1) — 상태 브리핑 재료, 매수/매도 의견 아님.
- 신규 ETF 관찰 후보 (PUSH 2) — 관찰 후보 알림, 진입 추천 아님.
- 급락 ETF 주의 신호 (PUSH 3) — 단순 임계값 기반 주의 신호.

위 3-PUSH 는:
- B 방향 전환 이후에도 **있는 그대로 유지**한다.
- 메인 출력이 정의되기 전까지는 **운영 의사결정의 1차 입력으로 사용하지 않는다**.
- 진입 시점 정량 분석은 시스템이 하지 않으며, 진입 여부 판단은 사용자가 AI
  투자세션에서 토론 후 결정한다.

## 7. 운영 원칙

- **PC 작업과 OCI 작업 분리**: PC 작업은 사용자가 직접 화면 앞에 앉아 진행하는
  발굴/판단 작업이고, OCI 작업은 사용자가 부재중일 때도 자동으로 도는 알림 작업이다.
- **PC 작업 빈도**: 주 2회 예상 (상한 아님). 사용자가 가능한 시점에 PC 앞에서
  발굴/판단 작업을 진행한다. 빈도는 운영하며 조정 가능.
- **OCI 작업 빈도**: 일 3회 자동 PUSH (장초 / 점심 / 장마감 전). OCI 작업은
  알림 배관 유지용이며 메인 의사결정 입력이 아니다.
- **PC 분석 평면 / OCI 운영·조회 평면 분리** (2026-06-20 추가, 본 문서
  `docs/handoff/PC_OCI_ARCHITECTURE_DIRECTION.md` 참조):
  - **PC = 분석·판단 평면**: 시장 데이터 SQLite 관리, ETF universe 갱신, 후보
    ETF 검토, ML 학습·백테스트·feature 실험, AI 투자세션, 사용자 승인, approved
    PARAM 과 published data snapshot 생성. PC 는 24시간 상시 실행을 전제로
    하지 않는다.
  - **OCI = 상시 운영·조회 평면**: latest approved PARAM 보관, 일 3회 3-PUSH
    실행, Telegram 발송. **OCI 장기 역할 — 외부 / 모바일에서 마지막 published
    데이터와 운영 상태를 조회할 수 있는 read-only 환경**으로 확장. OCI 는 ML
    학습을 수행하지 않는다.
  - **데이터 흐름**: PC SQLite 는 PC 작업용 기준 저장소로 유지. PC ML 이 OCI
    DB 를 직접 원격으로 읽지 않는다. PC 는 승인 / 발행 시점에 OCI 로 read-only
    published snapshot 을 전달한다 (구체 형식은 OCI 조회 메뉴 구현 직전 결정).
- **2주 미갱신 알림**: 어떤 산출물이든 2주 이상 갱신되지 않으면 사용자에게 알림.
  운영 단절이 길어졌음을 인지하기 위한 가드.
- **모바일 UI 후순위 — 유지** (2026-06-20 재확인): OCI 가 외부 / 모바일 조회
  평면으로 확장된다는 장기 방향에도 불구하고, 현재 단계의 모바일 UI 후순위
  원칙은 그대로 유지된다. 모바일 UI 는 PC 작업 1~7단계 + OCI 작업 8~11단계가
  충분히 검증된 뒤 12단계에서 다룬다. 현재 단계의 작업 대상 아님.
- **K6/EOD 기준**: 장중 실시간 반응 아님.
- **본업 우선**: 본업 지장 없는 수준에서만 운영.

## 운영 사이클

PC 작업 (주 2회 예상, 상한 아님):

1. universe 갱신 — 한국 상장 ETF universe / 수익률 / TOP N 후보 산출.
2. 후보 해석 — PC 앞에서 직접 후보 목록을 검토.
3. AI 투자세션 토론 — 후보를 AI 와 함께 해석.
4. 매매 결정 또는 보류 — 사용자가 명시적으로 결정.
5. 결정 기록 — 결정 사유 / 보류 사유를 기록 (운영 데이터 누적).
6. 다음 PC 작업 시점 결정.

OCI 작업 (일 3회 자동 PUSH):

7. 보유 종목 상태 브리핑 (보조 PUSH 1).
8. 신규 ETF 관찰 후보 (보조 PUSH 2).
9. 급락 ETF 주의 신호 (보조 PUSH 3).
10. 알림 수신 후 사용자가 PC 작업으로 넘어갈지 판단.

위 항목 중 **메인 의사결정 입력은 PC 작업 1~5단계** 다. OCI 작업은 보조 알림이며
**자동 매수/매도 실행은 어떤 단계에서도 수행하지 않는다**.

## 9.5. ML 방향성 — 2축 (2026-06-06 ETF Exposure Data Unfolding 1차 추가)

ML 방향성은 **두 축**으로 관리한다. 본 문서는 방향성 앵커이며, 구체 factor /
threshold / label / 모델 / 학습 데이터는 본 시점에 확정하지 않는다.

### 축 1. 상승 후보 발굴 / 점수화

- 현재 Market Discovery 흐름 (수익률 기반 TOP N, 초과수익, 구성종목, 중복률)
  의 연장선.
- 향후 ML 점수화 후보 — 후보 ETF 별로 단기/중기 점수를 매겨 사용자가 AI 와
  대화할 때 정량 재료로 사용.
- 본 프로젝트에서 **먼저 검증할 축** — 데이터 source 가 가장 가까이 있다.

### 축 2. 위험 감지 = 위험 구간 분류

- 사용자 목적에 더 가까운 축. 매수보다 손실 회피의 가치가 크다는 관찰.
- 표현 정의: **"하락 예측"이 아니라 "위험 구간 분류"** 다. 미래 하락을 미리
  맞히는 모델을 만들지 않는다. 시장 상태가 "위험 구간"인지 "정상 구간"인지
  분류하는 문제로 정의한다.
- 후보 지표 (factor 확정 X): 변동성, 거래량 급변, 외국인/기관 수급, 시장 폭,
  NAV/괴리율, 시장지수 시계열 등.
- **선행 조건**: 위 지표들의 시계열 적재. 단면 데이터로는 본 축을 다룰 수
  없다 — 시계열이 먼저다.
- threshold / factor / label / 학습창 / 모델 — 본 시점 미확정. 시계열 적재가
  먼저 끝난 뒤 사용자 결정 영역.

### 축 간 관계

- 두 축은 서로 독립이 아니다. 축 2 의 위험 구간 신호는 축 1 의 점수에 가중치
  로 들어갈 수 있다.
- 다만 본 시점에서는 축 1 / 축 2 를 분리해서 관리한다. 동시에 둘 다 손대지
