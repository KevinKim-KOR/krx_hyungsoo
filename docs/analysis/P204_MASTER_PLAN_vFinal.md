# P204 최종 MASTER PLAN vFinal (합의용)

## 1. 이번 P204의 한 줄 정의

**P204는 코어 백테스트 엔진을 뜯지 않고, `run_tune.py` 경계 안의 튜닝 평가층만 강화해서 “멀티 구간 평가 + 과최적화 억제 + 재시작 안전망 + 검산 가능한 리포트”를 갖춘 올웨더 튜닝 체계로 전환하는 세션이다.**

이 정의가 중요한 이유는 간단합니다.
핸드오프가 이번 세션 범위를 ML 튜닝으로 한정했고, UI도 `▶️ Run Tune`이 `run_tune.py`를 실행하고 `reports/tuning/tuning_results.json` 갱신을 성공 기준으로 보기 때문입니다.

## 3. 핵심 합의 사항 (GPT + Gemini + User)

### 합의 1. 공사 순서는 "인프라 먼저"가 맞다
중간 저장도 없는데 멀티 구간과 벌점 수식부터 넣으면, 한 번 뻗을 때 디버깅도 못 하고 증거도 날아갑니다.
그래서 **실제 구현 순서는 인프라 ➜ 구간 평가 ➜ 목적함수 ➜ 리포트**로 고정합니다.

### 합의 2. 구간 분할은 “안전한 분할”이 먼저다
외부 벤치마크나 추가 데이터에 기대면 코어 데이터 로더를 건드릴 위험이 있습니다.
* **Primary**: `nav_history` 기반 **기간 균등 분할** 또는 **분위수 분할**
* **Secondary**: 내부 데이터만으로 안정적으로 계산 가능한 경우에만 “상승/스트레스/횡보” 해석 레이어 추가
* **금지**: 외부 벤치마크 연동을 전제로 한 구간 판별

### 합의 3. OverfitPenalty 철학은 유지하되 Safe Math를 넣는다
* **벌점은 강하게 준다** ("한 구간이라도 처참하면 강하게 벌점")
* 하지만 **무한대/NaN/overflow가 안 나도록 safe math(clip/cap)로 구현한다.**
즉, 철학은 공격적으로, 계산은 보수적으로 갑니다.

## 4. 이번 P204의 설계 원칙 7개

1. **코어 엔진 비침습**: `nav_history`를 튜닝 래퍼에서 다룰 수 있으므로, **코어 엔진 수정 없이** 평가층에서 멀티 구간을 처리하는 것이 1순위.
2. **단일 기간 최적화 종료**: P204 이후의 점수는 반드시 **전체 기간 + 다중 구간**을 함께 반영해야 함.
3. **인프라 먼저**: SQLite, resume, atomic checkpoint가 먼저 기반이 되어야 함.
4. **수학은 Safe Math**: 지수 벌점, 비선형 벌점 허용하되 overflow/inf/NaN 방어 필수.
5. **결과는 사람도 읽을 수 있어야 함**: 특정 상위 후보의 일별 기록을 CSV로 내보내어 수동 검증 보조.
6. **UI 계약은 유지**: `▶️ Run Tune` 및 `▶️ Run Backtest`의 버튼/파일 경로 등은 변경 금지.
7. **운영 파이프라인 전제**: 토큰/승인/해시/Sync/Auto Ops는 완료된 운영 기반이며 절대로 건드리지 않음.

## 5. 최종 단계 구조 (마스터 플랜 순서)

### Step 0. 설계 계약 Freeze
실제 ко딩 전에 문서로 한계와 규칙 고정 (현재 이 문서).

### Step 1. 인프라 공사
“복잡한 평가를 얹기 전에, 튜닝이 죽어도 다시 살아날 수 있게 만든다.”
* Optuna study의 SQLite 영속화 및 이어받기(resume)
* Trial 단위 checkpoint 및 실패/중단 로그 보존 (1% atomic save)

### Step 2. 시간 자르기 (Multi-Slice)
“통짜 기간 점수 1개만 보는 눈먼 튜닝을 끝낸다.”
* 1회 백테스트 가동 후 `nav_history` 데이터를 사후 분할 (코어 비침습)
* 균등 3분할 혹은 분위수 분할 도입.

### Step 3. 목적함수와 벌점 체계
“높은 점수 1개가 아니라, 무난하게 강한 놈을 우승시키는 점수판을 만든다.”
* CAGR, MDD, Sharpe 종합 목적함수 생성
* 구간 간 편차 발생 및 극단적 무너짐 시 강한 Penalize 적용 (Safe Math 필수)

### Step 4. 결과 리포팅과 검산 자료
“튜닝 결과를 숫자 한 줄이 아니라, 검증 가능한 재료 묶음으로 만든다.”
* `tuning_results.json` 내역 확장 (Segment metrics, Penalty Breakdown 등)
* 상위 후보 파일 CSV 디버그 파편화 출력

### Step 5. 검증 루프
“돌아간다”가 아니라 “다시 돌려도 이어지고, 검산해도 말이 된다”를 확인한다.
* Quick Tune 1회 ➜ Resume 재실행 1회 ➜ `▶️ Run Backtest` 리-런 검산 ➜ CSV 육안 점검.
