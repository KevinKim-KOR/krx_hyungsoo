## 1. 시스템 정체성

### 1.1 친구 시스템

```text
친구 시스템:
사람의 경험/노하우로 점수체계 설계
→ 종목 풀 순위화
→ 백테스트 검증
→ 운영
```

### 1.2 사용자 시스템

```text
사용자 시스템:
AI 와 토론해 점수체계 초안을 만든다
→ 실제 운영한다
→ 운영 결과를 AI 와 함께 해석한다
→ 다음 변수와 점수체계를 조정한다
→ 필요 시 ML 로 결과를 분석한다
→ 다시 운영한다
```

### 1.3 핵심 차이

```text
차별화 포인트는 점수체계를 누가 처음 만들었느냐가 아니라,
운영 결과를 AI 와 함께 해석하고 다음 변수를 설정하는 반복 사이클이 있는가이다.
```

이것이 Q5 의 검증 대상이다 (§4.2 참조).

## 3. Momentum Engine 방향

### 3.1 결정

```text
외부 발굴과 보유 점검은 별도 로직이 아니라,
같은 Momentum Engine 을 서로 다른 데이터셋에 적용하는 구조로 간다.
```

### 3.2 구조

```text
Momentum Engine
├─ holdings mode
│  └─ 보유 종목 점검
└─ universe mode
   └─ 외부 ETF/섹터 후보 발굴
```

### 3.3 각 모드 목적

```text
holdings mode:
이미 보유한 종목이 아직 달리고 있는지 점검한다.

universe mode:
보유 여부와 무관하게 밖에서 잘 달리는 ETF/섹터 후보를 발굴한다.
```

### 3.4 두 모드의 공유점 / 분기점

- **공유점**: 점수체계 / 정렬 / 판단 사유 문장 생성 흐름은 같은 엔진을 공유한다.
- **분기점**: 입력 데이터셋(holdings vs universe), 결과를 어디에 표시하는가(승인 초안 vs 후보 발굴 화면), 어떤 메시지에 들어가는가(기존 승인 message_text vs 별도 알림) 는 다르다.
- **세부 분기 정책은 다음 구현 STEP 에서 결정한다** (Step4 에서 박지 않음).

## 5. 친구 시스템 참고 방식

### 5.1 참고할 것

```text
- 종목 풀을 순위화한다는 발상
- 점수와 근거를 함께 보여주는 구조
- 순위 변화가 중요하다는 점
- 사용자가 보기 편한 방식으로 시간을 들여 UI 를 다듬었다는 과정
- 여러 신호를 종합해 후보를 보는 방식
```

### 5.2 가져오지 않을 것

```text
- 친구의 점수 산식
- 친구의 컬럼 구성 그대로
- 친구의 UI 구조 그대로
- 친구의 종목 풀 그대로
- 친구의 threshold 그대로
- 친구의 운영 방식을 통째로
```

### 5.3 한 줄 정리

```text
친구에게서 배울 것은 결과물이 아니라 과정이다.
```

POC2-Step2D ~ Step3 종료까지 KS-3 비건설적 핑퐁 패턴이 재발하지 않았음을 근거로 ANSWERED. 단 "영원히 막히지 않는다" 는 의미가 아니며 새로운 단계에서 3턴 이상 산출물 없이 논쟁이 반복되면 KS-3 는 다시 발동할 수 있다 (남은 주의).

### 7.2 Q5 신규 등록 — OPEN

```text
Q5. AI 와 토론으로 점수체계를 만들고 운영하며 다듬는 방식이 친구의 경험 기반 방식보다
사용자 본인에게 더 적합한 투자 판단/운영 방식을 만드는가?
```

- **상태**: OPEN
- **검증 대상**: 사용자 본인
- **혼동 제거**: 와이프는 UI 가독성 검증 대상이며 투자 판단/운영 방식 적합성 검증 대상이 아니다 (PROJECT_ORIGIN_INTENT 의 성공 기준 3 — "와이프가 이해할 수 있는 UI" 와 분리).
- **데드라인**: 모멘텀 엔진 최소 운영 사이클 1회 완료 후 1차 검토.

### 7.3 Q4 OPEN 유지 — Step4 해석 보강

```text
Q4 는 OPEN 유지한다.
다만 Step4 기준 해석은 "잘 올라가는 섹터/ETF 발굴은 holdings 분석 factor 와
목적은 다르지만, 완전히 별도 시스템이 아니라 같은 Momentum Engine 의 universe mode 로
다룬다" 로 정리한다.
```

## 8. 다음 단계 진입 가드

다음 STEP (Momentum Engine 첫 구현 STEP) 으로 바로 들어가지 않는다.

다음 STEP 진입 전 필요한 결정 (Step4 에서는 답을 미리 박지 않음):

1. holdings mode 와 universe mode 중 어느 쪽을 먼저 구현할 것인가?
2. 첫 점수체계의 입력 변수는 무엇으로 시작할 것인가? (예: 1개 시작 / 여러 개 시작)
3. 데이터 소스는 Naver 시세 캐시만 재사용할 것인가, 아니면 신규 소스가 필요한가? (pykrx 등은 별도 사용자 승인 필요)
4. universe 종목 후보는 어떻게 결정할 것인가? (수기 ETF 목록 / 외부 데이터 / Phase 1 자산 재사용 등)
5. 결과를 어디에 표시할 것인가? (기존 승인 초안 흐름 재사용 / 별도 화면 신설 / 별도 message_text 등)
6. ML 연결 시점은 언제인가? (현재는 정적 점수체계만, ML 은 사이클 안정 후로 보류)

위 6개 질문은 다음 STEP 의 설계 지시문에서 새로 결정한다.

## 9. 절대 쓰지 않을 표현 / 행위

다음 STEP 에서 본 설계서를 잘못 해석하지 않도록 가드:

```text
- 다음 단계에서 BUY/SELL 추천 구현
- 다음 단계에서 리밸런싱 구현
- 다음 단계에서 ML 바로 구현
- 다음 단계에서 친구 UI 복제
- 다음 단계에서 모든 점수 산식 / threshold / 유니버스 / 데이터 소스 한 번에 결정
- 다음 단계에서 holdings mode 와 universe mode 동시 구현 강제
- "잘 달리는 말 찾기" 가 완료되었다는 표현
- "투자 판단 엔진 완성" 이라는 표현
- "ML 확장 검증 완료" 라는 표현
```

본 설계서가 고정하는 것은 오직 다음이다:

```text
- Momentum Engine 의 두 모드 (holdings / universe) 구분 방향
- 친구 시스템에서 가져올 것과 가져오지 않을 것의 구분
- 시각화 최소 유지 / 본격 시각화 보류 정책
- Q3 ANSWERED, Q5 OPEN 신규, Q4 OPEN 유지
- 구체 산식·유니버스·데이터 소스·ML·UI 는 다음 STEP 에서 결정
```

## 10. 최종 선언

POC2-Step4 는 **방향 결정 설계 문서 저장 단계** 다.

다만 다음을 잊지 않는다.

```text
이 단계는 구현 단계가 아니다.
이 단계는 점수 산식을 박는 단계가 아니다.
이 단계는 ML 진입 단계가 아니다.
이 단계는 "잘 달리는 말 찾기" 완료 단계도 아니다.
이 단계는 Momentum Engine 의 두 모드 구분과 Q4 경계 / Q5 검증 대상을 박아두는 단계다.

다음은 위 §8 의 6개 질문에 답하는 첫 구현 STEP 의 설계 지시문 작성이다.
```
