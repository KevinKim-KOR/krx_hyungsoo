## 2. Step8 의 단일 목표

3-PUSH 최소 구조를 **실제 운영 흐름으로 반복 실행**하고, **사용자가 받은 PUSH 가 해석
가능하며 행동 또는 명시적 보류 판단으로 이어지는지 기록**한다.

### "행동" 의 정의 (실제 매수/매도만 의미하지 않음)

다음 모두 행동으로 본다:
- AI 투자세션으로 가져감
- 매수 결정
- 매도 결정
- **보류 결정** (= 명시적 보류 판단)
- 무시 결정
- 후보군 수정 필요 판단
- 메시지 이해 실패 판단
- 운영 절차가 번거롭다는 판단

→ 사용자의 모든 명시적 반응이 "행동" 으로 기록되며 검증 데이터가 된다. 침묵 / 미응답은
행동이 아니다.

## 3. Step8 완료 판단 기준

### 3.1 표준 완료

다음을 **모두** 만족하면 Step8 표준 완료:

1. 서로 다른 영업일 기준 **3회 이상** 운영 기록이 docs/ops/POC2_STEP8_OPERATION_LOG.md
   에 남는다.
2. 각 운영 기록에 PUSH 1 / PUSH 2 / PUSH 3 상태가 기록된다.
3. 각 운영 기록에 **사용자의 판단** 이 기록된다 (다음 중 하나 이상):
   - AI 토론 필요 / 매수 / 매도 / 보류 / 무시 / 이해 불가 / 개선 필요
4. **최소 1회 이상** AI 투자세션으로 가져갔거나, 가져가지 않은 이유가 명시된다.
5. 사용자가 "**이 운영 사이클은 계속 돌릴 수 있다 / 없다**" 를 판단한다.

### 3.2 조기 종료

3회 운영 전이라도 다음 중 하나가 발생하면 Step8 을 **조기 종료** 할 수 있다:

- 사용자가 메시지를 이해할 수 없다고 판단
- 수동 운영 절차가 너무 번거롭다고 판단
- Telegram 메시지가 의사결정에 전혀 도움이 되지 않는다고 판단
- 알림이 과하거나 부족하다고 판단
- seed / refresh / draft / approve / Telegram 중 운영 병목이 재발

**조기 종료는 실패가 아니라 운영 검증 결과로 기록한다.** 운영 로그 문서에 사유와
다음 단계 후보를 명시한다.

### 3.3 첫 의사결정 도달 (Q5 BACKLOG 복귀 핵심 트리거)

다음 흐름이 발생하면 Step8 의 **핵심 검증은 충족된 것** 으로 본다:

```
시스템 PUSH 수신
  → AI 투자세션 토론
  → 매수 / 매도 / 보류 중 하나를 명시적으로 결정
```

이 경우 **ASSUMPTIONS L-3 (이전 Q5) BACKLOG 복귀 조건이 충족** 된다 (§7).

## 4. 운영 방식

### 4.1 자동화 정책

- **자동 scheduler / cron 아님** — 운영은 사용자 명시 실행으로만 진행.
- K6/EOD 저빈도 원칙 유지 (PROJECT_ORIGIN_INTENT.md §7) — 장중 실시간 반응 아님.
- 직장인형 저빈도 — 본업 우선.

### 4.2 1회 운영 흐름 (참고)

```
EOD 시점에 사용자가:
  1. (필요 시) 신규 ETF 관찰 후보 갱신 클릭 → POST /universe/momentum/refresh
  2. 보유 종목 초안 생성 클릭 → POST /runs/generate-from-holdings
  3. Telegram PUSH 수신 (보유 종목 상태 브리핑 + 신규 ETF 관찰 후보 + 조건부 급락 신호)
  4. 메시지 해석 → 필요 시 AI 투자세션 진행 → 사용자 판단
  5. docs/ops/POC2_STEP8_OPERATION_LOG.md 에 1회 운영 기록 작성
```

**모든 단계는 사용자 명시 실행** — 자동화된 부분은 없다.

## 6. ASSUMPTIONS / BACKLOG 연결

### 6.1 Q4 (잘 올라가는 섹터/ETF 발굴 단위) — OPEN 유지

Step8 에서 수집할 Q4 데이터:

- 신규 ETF 관찰 후보가 실제로 관심을 끌었는가
- 신규 ETF 관찰 후보가 보류 / 무시로 끝났는가
- ETF 개별 후보만으로 충분했는가
- 테마/섹터 관점이 필요하다고 느꼈는가
- starter seed 가 너무 좁다고 느꼈는가
- 자동 ETF universe 수집 필요성이 실제로 생겼는가

**Step8 은 Q4 를 닫지 않는다** — 운영 데이터 수집만 한다. 닫는 시점은 운영 첫 달 종료
또는 사용자 명시 결정 시점.

### 6.2 Q5 (AI 토론 점수체계 검증) — BACKLOG

Q5 는 현재 ASSUMPTIONS L-3 항목으로 BACKLOG 이관 상태.

**복귀 조건** (ASSUMPTIONS.md L-3):
- 운영 시작 후 시스템 PUSH 를 받고,
- AI 투자세션을 거쳐
- 첫 매매 결정 또는 명시적 보류 결정 1회가 발생할 때.

**Step8 도중 이 조건이 충족되면 Q5 는 ASSUMPTIONS 활성 질문으로 복귀**.

복귀 시점 (다음 중 하나):
- 매수 결정 1회
- 매도 결정 1회
- 명시적 보류 결정 1회

**단순히 Telegram 을 받기만 한 것은 Q5 복귀 조건이 아니다.**

### 6.3 와이프 UI 검증 — BACKLOG 유지

복귀 조건:
- 사용자 본인이 운영 사이클을 반복 가능하다고 판단한 시점.

Step8 종료 시 사용자가 "이제 내가 운영 흐름은 이해한다" 고 판단하면, 와이프 UI 검증을
복귀 후보로 올릴 수 있다.

PROJECT_ORIGIN_INTENT.md §4 의 장기 성공 기준 "와이프가 이해할 수 있는 UI" 는 삭제하지
않는다 — 검증 시점만 미루는 결정.

## 7. 성공 / 부분 성공 / 실패 기준

### 성공 (Step8 표준 완료)

- 3회 이상 운영 기록 + AI 투자세션 1회 이상 + 사용자 운영 가능성 판단 명시.
- 추가로 첫 의사결정 (매수/매도/보류) 도달 시 → Q5 BACKLOG 복귀 트리거 동시 충족.

### 부분 성공

- 1~2회 운영 기록만 있으나 사용자가 운영 가능성을 판단하고 회고를 남긴 경우.
- 또는 첫 의사결정 도달했으나 운영 횟수 3회 미만.

### 실패 (= 조기 종료의 다른 표현)

§3.2 조기 종료 사유 중 하나 이상 발생.

- 메시지 이해 불가 / 운영 절차 번거로움 / Telegram 의사결정 무가치 / 알림 빈도 부적절 /
  운영 병목 재발.
- 실패 자체가 검증 결과 — 다음 STEP 의 입력 데이터가 된다.
