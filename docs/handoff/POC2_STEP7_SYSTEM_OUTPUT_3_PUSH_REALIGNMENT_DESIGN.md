## 1. Step7 의 성격

- **구현 단계가 아니다.** 본 STEP 은 코드 / API / UI / Telegram / 데이터 계약을
  변경하지 않는다.
- 다음 구현을 위한 **구조 재정렬 설계 단계** 다.
- 기존 Step3 ~ Step6 산출물을 **버리지 않고**, 새 3-PUSH 구조에 맞춰 재배치한다.
- 본 설계서는 후속 구현 Step 의 진입 기준을 정한다 — 후속 Step 은 PUSH 별로 쪼개서
  하나씩 진입한다 (§13 후속 Step 분리 원칙).

## 2. 공식 PUSH 명칭 (중립형 — 매매 의견 드리프트 차단)

기존 표현 "보유 종목 추가매수/매도 의견", "신규 ETF 추천" 은 시스템 출력의 공식
명칭으로 **사용하지 않는다**. 시스템이 매매 결정을 내리는 것으로 오해될 위험이
크기 때문이다 (레드팀 MINOR 지적).

### 공식 명칭 3가지

| 번호 | 공식 명칭 | 의미 한정 |
|---|---|---|
| **PUSH 1** | **보유 종목 상태 브리핑** | 현재 보유 종목 상태 정보 제공 (매매 의견 아님) |
| **PUSH 2** | **신규 ETF 관찰 후보** | pykrx 기준 모멘텀 상위 ETF 1건 관찰용 (매수 추천 아님) |
| **PUSH 3** | **급락 ETF 주의 신호** | 급락 ETF 알림 (매도 지시 아님, 자리만 잡음) |

이 명칭은 PROJECT_ORIGIN_INTENT.md §3 "시스템 출력 정의" 와 일관되며, 본 설계서
이후 모든 문서 / 코드 주석 / 핸드오프에서 위 공식 명칭을 사용한다.

## 3. 투자 행동 지시 금지 원칙

본 시스템의 책임 경계를 다음과 같이 못 박는다.

- **시스템은 매수/매도 결정을 내리지 않는다.**
- 시스템은 **판단 재료와 관찰 후보를 제공**한다.
- **최종 매매 결정은 사용자**가 한다.
- **AI 투자세션은 해석 보조 채널**이지 결정자가 아니다.
- **PUSH 1 은 향후에도 매수/매도 의견 자동화가 아니다.**
- **PUSH 2 는 매수 추천이 아니다.**
- **PUSH 3 은 매도 지시나 매수 금지 명령이 아니다.**

본 원칙은 코드 주석 / message_text 문구 / Telegram 발송문 어디에서도 위반하지 않는다.
"BUY" / "SELL" / "리밸런싱" / "매수 신호" / "매도 신호" 같은 표현은 시스템 출력에
넣지 않는다.

## 4. Step6 의미 제한

POC2-Step6 의 결과는 **다음 의미로만 기록**한다.

- ETF 개별 후보 모멘텀 점검 배관 검증 완료
- pykrx 1개월 수익률 계산 성공
- universe_momentum_latest.json 저장 성공
- top_candidate 1개가 message_text / Telegram 판단 사유 3번째 bullet 까지 도달
- Telegram 수신 확인 완료 (사용자 운영 테스트 — 2026-05-11)

Step6 을 다음처럼 해석하거나 표현하지 않는다 (드리프트 방지):

- ✗ 섹터/테마 발굴 완료
- ✗ 신규 ETF 매수 추천 체계 완성
- ✗ 보유 종목 매수/매도 판단 체계 완성
- ✗ 급락 ETF 경고 구현
- ✗ ML 검증 완료

## 5. 기존 산출물의 3-PUSH 매핑

Step3 ~ Step6 산출물을 새 3-PUSH 구조에 다음과 같이 매핑한다 — 기존 코드 / 데이터
계약은 보존하고 분류만 재정렬한다.

| 기존 산출물 | 매핑되는 PUSH | 역할 |
|---|---|---|
| **Step3 보유 비중 영향** (factor_signals scope=portfolio) | PUSH 1 보유 종목 상태 브리핑 | 재료 |
| **Step5B holdings momentum** (draft_payload.momentum_result) | PUSH 1 보유 종목 상태 브리핑 | 재료 |
| **Step6 universe momentum** (factor_signals scope=universe + universe_momentum_latest.json) | PUSH 2 신규 ETF 관찰 후보 | 재료 |
| **급락 ETF 경고** (현재 구현 없음) | PUSH 3 급락 ETF 주의 신호 | 자리만 잡음 |

위 매핑은 본 Step7 의 결과물이다. 후속 구현 Step 진입 전 본 매핑 표를 변경하려면
새 데이터/근거를 ASSUMPTIONS 또는 PROJECT_ORIGIN_INTENT 에 기록 후 변경한다
(KS-11 의사결정 24시간 룰 적용 — §8).

## 6. PUSH 별 처리 범위

### PUSH 1 — 보유 종목 상태 브리핑

- 현재 보유 종목의 **상태 정보**를 제공한다.
- 현재의 "보유 비중 영향", "모멘텀 점검" 은 **매매 의견이 아니라 상태 브리핑 재료** 다.
- **무릎/머리/어깨 기준은 확정하지 않는다.**
- 매매 의견 드리프트 차단 — 정확한 문구로 다음과 같이 명시한다:

> "PUSH 1 은 매수/매도 의견으로 구현하지 않는다.
> 향후에도 시스템 출력은 보유 종목 상태 브리핑과 판단 재료 제공에 머문다.
> 최종 매수/매도 판단은 사용자와 AI 투자세션의 해석 영역으로 둔다."

### PUSH 2 — 신규 ETF 관찰 후보

- Step6 universe momentum 결과를 이어받는다.
- pykrx 1개월 수익률 기준 **top_candidate 1개를 관찰 후보**로 보여주는 구조다.
- **매수 추천이 아니다.**
- 후보 전체 목록을 Telegram 에 보내지 않는다 (Step6 §13 / AC-28 가드 유지).
- **자동 ETF universe 수집은 하지 않는다.** universe seed 는 사용자가 manual 로 입력.
- manual seed JSON 입력 불편은 **운영 UX BACKLOG** 로 둔다 (§9 BACKLOG 후보 1번 항목).

### PUSH 3 — 급락 ETF 주의 신호

- 3번째 PUSH 유형으로 **자리만 잡는다.**
- **현재 구현은 없다.**
- **급락 임계값은 확정하지 않는다.**
- 급락 계산 / Telegram 발송 / 알림 정책은 **후속 별도 Step** 으로 분리한다 (PUSH 3 구현 Step).

## 7. 운영 빈도

Step7 설계서는 **장초 / 점심 / 장마감 전 3회 분석을 기정사실화하지 않는다.**

PROJECT_ORIGIN_INTENT.md §7 운영 원칙과 충돌 소지가 있으므로 본 설계서는
**문서 간 정합성 보정 대상** 으로 분류 (§9 BACKLOG 후보 "운영 빈도 문서 정합성 보정").

본 Step7 의 기본 운영 기준:

- **K6/EOD 기준**
- **장중 실시간 반응 아님**
- **직장인형 저빈도 운영**

EOD 기준의 운영 사이클은 다음처럼 단순화 표현한다:

> "EOD 분석 후:
> - 보유 종목 상태 브리핑 확인
> - 신규 ETF 관찰 후보 확인
> - 급락 ETF 주의 신호 확인
> - 필요 시 AI 투자세션에서 토론
> - 사용자가 매매 결정 또는 보류"

장초 / 점심 / 장마감 전 3회 분석 vs EOD 저빈도 — 두 표현은 **문서 정합성 보정 작업**
에서 별도로 통합한다. 본 Step7 에서 확정하지 않는다.

## 8. KS-11 (의사결정 24시간 룰) 요약 — Step7 적용

KILL_SWITCHES.md KS-11 의 요약을 본 설계서 안에 포함한다:

- **증상**: 새 데이터 없이 기존 결정을 24시간 안에 뒤집기.
- **조치**: 결정 변경 시 새 데이터/근거를 ASSUMPTIONS 또는 PROJECT_ORIGIN_INTENT 에
  기록 후 변경.
- **적용 대상**: 운영 빈도, 발굴 단위, 점수체계 기준 등 시스템 방향 결정.

### Step7 에서 KS-11 적용 대상 (= 본 STEP 에서 변경 금지)

본 Step7 에서 다음 항목들은 **변경하지 않는다**. 변경하려면 새 데이터/근거 기록 후
별도 STEP 으로 처리한다.

- 운영 빈도 변경
- 발굴 단위 변경
- 기간 기준 변경
- 급락 임계값 도입
- 보유 종목 상태 브리핑을 매매 의견으로 승격
- 신규 ETF 관찰 후보를 매수 추천으로 승격

## 9. BACKLOG 후보

본 Step7 에서 도출되었으나 본 STEP 에서 처리하지 않고 BACKLOG 로 분리하는 항목들.
각 항목은 docs/backlog/BACKLOG.md 에 등록한다.

### 9.1 manual seed 입력 UX 개선
- 보류 사유: Step6 의 manual universe seed JSON 입력은 사용자에게 작성 부담이 큼.
  그러나 자동 universe 수집은 §6 의 "자동 ETF universe 수집은 하지 않는다" 정책에
  막혀 있어, UX 개선은 별도 작업 영역.
- 보류된 위험: 운영 시작 직후 사용자가 universe seed 작성에 진입 장벽을 느껴 PUSH 2
  사이클 자체가 시작되지 못할 가능성.
- 복귀 조건: PUSH 2 구현 Step 진입 시 또는 사용자가 명시적으로 입력 UX 부담을
  보고할 때.

### 9.2 기본 ETF starter seed
- 보류 사유: 사용자 입력 부담 완화용 starter seed 가 임의 추천으로 해석될 위험.
  "기본 추천 ETF" 가 시스템의 매수 추천처럼 오해될 수 있음.
- 보류된 위험: starter seed 의 종목 선택이 매수 추천으로 굳어지면 §3 투자 행동
  지시 금지 원칙과 충돌.
- 복귀 조건: 9.1 manual seed 입력 UX 개선 검토 시점에 starter seed 를 "예시 목록"
  으로 한정해서 함께 검토.

### 9.3 급락 ETF 주의 신호 최소 구현 (PUSH 3 구현 Step)
- 보류 사유: §6 PUSH 3 는 자리만 잡고 구현 없음. 급락 임계값 확정 필요.
- 보류된 위험: 너무 민감하면 알림 과다 (KS-5), 너무 둔하면 경고 실패.
- 복귀 조건: PUSH 1 또는 PUSH 2 구현 Step 1회 이상 완료 후 사용자가 PUSH 3 진입 결정.

### 9.4 무릎/머리/어깨 정량 기준
- 보류 사유: PUSH 1 의 핵심 기준이지만 백테스트/운영 데이터 없음.
- 보류된 위험: 임의 기준을 쓰면 §3 의 "매매 의견 자동화 아님" 원칙과 충돌.
- 복귀 조건: PUSH 1 구현 Step 진입 + 백테스트 또는 운영 데이터 1개월 누적.

### 9.5 기간 비교 정책 (1일 / 1개월 / 60일 / 120일 / 3개월)
- 보류 사유: Step6 은 pykrx 1개월 수익률 1개만 사용. 다른 기간 비교 필요성은 운영
  결과로 결정.
- 보류된 위험: 기간 임의 추가는 친구식 복합 점수체계 회귀 위험 (§9.6 Layer C 와 연결).
- 복귀 조건: 1개월 외 기간 비교 필요성이 운영 중 명시 발생 또는 백테스트 설계 시점.

### 9.6 RS / 거래량 / 정배열 등 복합 지표
- 보류 사유: 친구식 복합 점수체계로 회귀할 위험.
- 보류된 위험: 단순한 시스템을 만들기 전에 복합 지표가 들어오면 해석성과 운영성이 무너짐.
- 복귀 조건: 단일 가격 기반 변수 운영 결과가 부족하다는 증거 + 백테스트 인프라 준비.

### 9.7 운영 빈도 문서 정합성 보정
- 보류 사유: PROJECT_ORIGIN_INTENT §7 의 "1일 3회 PC 분석" 과 본 Step7 의 "K6/EOD
  저빈도" 표현이 충돌 소지.
- 보류된 위험: 두 표현이 공존하면 후속 STEP 설계에서 운영 빈도 결정이 흔들림.
- 복귀 조건: PUSH 1 / 2 / 3 중 첫 구현 Step 진입 직전 문서 정합성 보정 작업 1회.

  > "PUSH 1 은 매수/매도 의견으로 구현하지 않는다. 향후에도 시스템 출력은 보유 종목
  > 상태 브리핑과 판단 재료 제공에 머문다. 최종 매수/매도 판단은 사용자와 AI
  > 투자세션의 해석 영역으로 둔다."
- "아직 매수/매도 의견으로 구현하지 않는다" 라는 미래 시제 표현은 사용하지 않는다
  (드리프트 유발 가능 — "아직 = 향후엔 가능" 으로 해석될 위험).

## 13. 후속 Step 분리 원칙

본 Step7 의 결정으로, 후속 구현 STEP 은 다음 원칙을 따른다:

1. **한 번에 하나의 PUSH 만 다룬다.** PUSH 1 / 2 / 3 를 동시에 진입하지 않는다.
2. 각 PUSH 구현 STEP 은 본 설계서 §6 의 처리 범위 안에서만 작업한다.
3. 본 설계서 §10 의 AC 위반이 발생하면 즉시 STEP 진입 중단 + 본 설계서 재검토.
4. PUSH 1 구현 시 무릎/머리/어깨 기준이 필요하면 §9.4 BACKLOG 항목 복귀 + 별도
   백테스트/운영 데이터 누적 후 진입.
5. PUSH 3 구현 시 급락 임계값 필요 시 §9.3 BACKLOG 항목 복귀 + 별도 임계값 결정
   작업 선행.
