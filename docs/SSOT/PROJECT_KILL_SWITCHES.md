# KILL_SWITCHES.md

작성 목적: 프로젝트 전체에서 즉시 멈춰야 하는 상황만 최소한으로 고정한다.
성격: project-level kill switches
우선순위: 최상
변경 빈도: 낮음

---

## 1. 공통 원칙

아래 중 하나라도 발생하면 현재 논의나 진행을 즉시 보류 또는 중단한다.
이 문서는 모든 스텝에 공통으로 적용되는 최소 브레이크다.

---

## 2. 증거 정합성 중단 조건

### KS-P1. stale 세트
- UI / evidence md / result json / compare summary / handoff가 같은 run / asof / timestamp를 가리키지 않으면 중단
- 조치: stale 경고 후 canonical 기준 재정렬

### KS-P2. handoff가 새 truth처럼 동작
- handoff mirror가 canonical을 대체하거나, UI가 handoff를 직접 truth source처럼 읽기 시작하면 중단
- 조치: handoff는 전달용으로만 제한

### KS-P3. Main Run / Compare Run 혼용
- 대표 결과와 비교 실험군 숫자를 섞어 설명하면 중단
- 조치: 기준선 구분부터 다시 고정

---

## 3. 구조/범위 중단 조건

### KS-P4. closeout 축 재개
- 이미 닫힌 축을 wording / threshold / mode 수준으로 다시 파기 시작하면 중단
- 조치: same-axis micro-tuning으로 판정

### KS-P5. 여러 병목 혼합
- 한 스텝에서 timing / allocation / holding / selection / UX / ML / source extraction을 동시에 다루면 중단
- 조치: 병목 하나만 남기고 분리

### KS-P6. friend source 통째 이식
- 규칙 추출이 아니라 플랫폼 / 구조 / 계정 방식까지 통째로 옮기려 하면 중단
- 조치: 이식 금지

---

## 4. 운영 원칙 위반 중단 조건

### KS-P7. 직장인형 저빈도 운영 위반
- 장중 반응형 / 상시 관측형 / 실시간 외부 데이터 필수 구조면 중단

### KS-P8. 인간 승인 구조 부재
- AI가 추천만 하고 인간의 채택 / 보류 / 기각 기록 구조가 없으면 중단

### KS-P9. shadow/live-lite 없이 곧바로 실운영
- 운영 리허설 없이 바로 라이브 반영을 논하면 중단

---

## 5. AI 협업 중단 조건

### KS-P10. 설계자/레드팀/조언자 역할 혼선
- 설계자가 스스로 레드팀 역할까지 하고,
- 레드팀이 설계자처럼 최종 설계를 확정하고,
- 조언자가 근거 없이 방향을 뒤집으면 중단
- 조치: 역할 정의부터 재고정

### KS-P11. 레드팀의 과잉 경직화
- 레드팀이 치명적 결함 지적은 하되, 판단 보조용 대안 방향 제시까지 전면 금지되는 구조면 중단
- 조치: "대안 방향 제시 가능, 최종 설계 확정 금지"로 수정

### KS-P12. 문서 과잉 증식
- 규칙을 늘려서 불안을 덮기 시작하면 중단
- 조치: 불변 레이어와 step 레이어를 분리하고 최소화
