# ASSUMPTIONS_UNDER_QUESTION.md

작성 목적: 프로젝트 전체에서 장기적으로 흔들리고 있는 가정을 분리 보관한다.
성격: project-level assumptions tracker
우선순위: 높음
변경 빈도: 중간

---

## 1. 사용 규칙

이 문서의 항목은 확정 사실이 아니다.
상태는 아래 4개만 쓴다.

- OPEN
- WEAK
- STRONGER_BUT_NOT_PROVEN
- REJECTED

새 가정이 생기면 step 본문보다 먼저 이 문서에 올린다.

---

## 2. 장기 가정 목록

### A-P1. 현재 KRX 엔진 구조 안에서 MDD 10% 미만 달성이 가능하다
- 상태: WEAK
- 이유: Phase 1 closeout 결과 동일 축 조정만으로 벽을 넘지 못했다.
- 주의: 같은 축 미세조정만으로 가능하다고 단정 금지

### A-P2. selection / filter 축만 더 다듬으면 병목이 풀린다
- 상태: REJECTED
- 이유: Track A / Track B closeout 결과로 설득력 약화
- 주의: 동일 축 재개 명분으로 재사용 금지

### A-P3. friend source 코드 규칙이 현재 엔진에 의미 있는 입력원이 될 수 있다
- 상태: OPEN
- 이유: 추출 가치는 있으나, 실제 효과와 이식 범위는 아직 미확정
- 주의: friend가 전업이라는 이유로 자동 우대 금지

### A-P4. friend의 진짜 edge가 코드 안에 충분히 담겨 있다
- 상태: WEAK
- 이유: 뉴스 해석, 상황 판단, 전업 운영 리듬 같은 코드 밖 요소가 큼
- 주의: 코드와 인간 판단 입력을 섞지 말 것

### A-P5. 더 큰 ML / GPU가 지금 병목을 해결할 수 있다
- 상태: WEAK
- 이유: 현재 병목은 연산량보다 문제 정의와 입력원 부족에 가까움
- 주의: Track B 재개 명분으로 사용 금지

### A-P6. 현재 UI를 거의 그대로 두고도 운영형으로 갈 수 있다
- 상태: REJECTED
- 이유: 연구용 누적 화면과 운영용 1페이지는 역할이 다름
- 주의: 연구 화면과 운영 화면 분리 필요

### A-P7. handoff만 잘 만들면 흐름이 자동 보존된다
- 상태: WEAK
- 이유: handoff는 전달 수단이지 안전판이 아님
- 주의: canonical과 handoff를 절대 혼동하지 말 것

### A-P8. 직장인형 운영에서도 friend의 운영 방식을 거의 그대로 쓸 수 있다
- 상태: REJECTED
- 이유: 전업과 직장인의 운영 리듬이 다름
- 주의: friend source는 번역해서 가져와야 함

### A-P9. 프로젝트의 최종 목적은 MDD 10% 미만 달성 하나로 충분하다
- 상태: WEAK
- 이유: 운영 가능성, 설명 가능성, shadow/live-lite 적합성도 함께 봐야 함
- 주의: 숫자 하나만 좋아도 성공이라는 프레이밍 금지

---

## 3. 현재 가장 중요한 장기 질문 Top 3

1. friend source가 실제로 새 입력원이 될 수 있는가
2. friend의 edge 중 코드 밖 요소를 어떻게 따로 다룰 것인가
3. 성능 목표와 운영 목표를 어떤 순서로 정리할 것인가
