# ORIGIN_INTENT.md

작성 목적: 프로젝트가 다시 "좋아 보이는 말 수집함"으로 흐르지 않도록, 바뀌지 않는 원칙만 고정한다.
성격: project-level anchor
우선순위: 최상
변경 빈도: 매우 낮음

---

## 1. 이 문서의 역할

이 문서는 프로젝트 전체의 불변 앵커다.
새로운 스텝, 새로운 후보, 새로운 문서가 생겨도 먼저 이 문서에 충돌하는지 확인한다.
이 문서와 충돌하는 제안은 설득 대상이 아니라 보류 대상이다.

이 문서는 "무엇을 더 할까"보다 "무엇을 하지 않을까"를 먼저 고정한다.

---

## 2. 프로젝트의 원래 목적

이 프로젝트의 목적은 단순히 가장 높은 수익률을 찾는 것이 아니다.
아래 4개를 동시에 만족하는 **직장인형 투자 운영 시스템**을 만드는 것이다.

1. **UI에서 재현 가능한 검증 체계**
   - 같은 run / asof / timestamp / experiment_id가 맞는 evidence 세트로 판단할 수 있어야 한다.
   - Main Run / Compare Run / analysis-only를 UI에서 설명할 수 있어야 한다.
   - canonical report와 handoff mirror를 혼동하지 않는다.

2. **사람 + AI 의사결정 루프**
   - AI는 자동매매 엔진이 아니라 설명·비교·회고 파트너다.
   - 인간이 최종 채택 / 보류 / 기각을 남길 수 있어야 한다.

3. **직장인형 저빈도 운영 가능성**
   - 장중 대응이 아니라 K6 / EOD 기준의 저빈도 운영을 전제로 한다.
   - 본업을 깨지 않는 짧은 체크포인트와 명확한 상태 요약이 있어야 한다.

4. **실운영으로 이어질 수 있는 구조**
   - 연구용 비교표와 문서화로 끝나지 않고,
   - shadow / live-lite 리허설로 이어질 수 있어야 한다.

---

## 3. 핵심 운용 철학

### 3.1 인간 최종 승인
- 인간이 이해 못 하는 블랙박스 규칙은 운영 원칙에 맞지 않는다.
- AI는 설명을 돕고, 최종 판단은 인간이 남긴다.

### 3.2 반증 가능성
- 말로만 통과하는 기준은 금지한다.
- 좋은 아이디어처럼 보여도, 나중에 틀렸다고 말할 수 없는 구조면 채택하지 않는다.

### 3.3 K6 / EOD 기반
- 상시 관측형, 장중 반응형, 실시간 뉴스 의존형 운영은 기본 원칙에 맞지 않는다.
- 직장인형 루프를 깨는 설계는 처음부터 보류한다.

### 3.4 source-derived 와 overlay 분리
- friend source에서 직접 읽힌 것은 core primitive로 적는다.
- 우리가 덧붙인 운영 규칙·트리거·정책은 overlay로 분리한다.
- 둘을 섞으면 실패 원인 추적이 흐려진다.

---

## 4. 이 프로젝트가 아닌 것

이 프로젝트는 아래를 목표로 하지 않는다.

1. 복잡한 ML 자체를 목적화하는 프로젝트
2. GPU 사용량으로 우수함을 증명하려는 프로젝트
3. 친구 프로젝트를 통째로 복제하는 프로젝트
4. 프론트엔드/저장소 구조를 통째로 갈아엎는 프로젝트
5. 실시간 외부 API 의존을 강화하는 프로젝트
6. 근거 없는 규칙을 계속 추가하는 프로젝트
7. 인간 뉴스 해석·veto 규칙을 섣불리 코드화하는 프로젝트

## 5. 현재까지의 교훈

- Phase 1은 current engine validation closeout 단계까지 진행되었다.
- Track B는 CLOSED_REJECTED다.
- 현재 엔진은 MDD 10% 벽을 넘지 못했다.

이 결론은 "프로젝트 실패"가 아니라,
**현재 엔진의 한계가 어디까지인지 확인한 것**으로 해석한다.

따라서 Phase 2는 Phase 1을 뒤엎는 단계가 아니라,
**friend source / 운영 UX / human+AI loop를 overlay 하는 단계**다.

---

## 6. 프로젝트 절대 기준

1. Main Run / Compare Run / analysis-only를 절대 섞지 않는다.
2. canonical report와 handoff mirror를 절대 혼동하지 않는다.
3. closeout 된 축은 같은 축 미세조정으로 다시 열지 않는다.
4. friend source는 규칙 / UX / 인간 판단 입력으로 분해해서 본다.
5. 새 제안은 먼저 "직장인형 운영과 충돌하는가"를 본다.
6. 새 가정은 본문보다 먼저 ASSUMPTIONS 문서에 올린다.

---

## 7. 현재 프로젝트 위치

- Phase 1: closeout 완료
- Track B: CLOSED_REJECTED
- 현재 진입점: Phase 2

즉 지금은 "같은 축을 또 미세조정"하는 단계가 아니라,
**friend source 추출 → 운영형 UX → human+AI decision loop 설계**로 넘어가는 단계다.
