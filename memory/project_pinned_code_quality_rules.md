# 코드 품질 / 구조 품질 / 문서 관리 필수 준수사항

이 프로젝트의 모든 작업에서 **무조건** 준수할 규칙. 다른 어떤 가이드보다 우선한다.
이전 세션에서 P207/P208/P209 god module 사고와 반복적 과장 보고 사고 이후
사용자가 명시적으로 고정한 규칙이다.

## 1. 기본 커뮤니케이션

1. 모든 답변, 계획, 보고, 문서는 **한국어** 로 작성한다.
2. 사용자의 질문에 답할 때는, 코드를 보며 수정할 부분을 발견해도 **즉시 수정하지 말고**
   먼저 질문에 답한 뒤 문제점을 보고하고 승인을 받아 수정한다.

## 2. 작업 절차

3. `znotes/` 폴더는 사용자 개인 메모이므로 **명시적 지시 없이는 읽거나 참조하지 말 것**.
4. 데이터 처리/연산은 **순차 처리 원칙**. 불필요한 병렬 처리 금지.
   (다수 백테스트를 수행하는 튜닝 작업만 예외 허용)
5. 리팩토링/구조 변경 시 `Dockerfile`, 배포 스크립트, 실행 경로 영향을 반드시 확인할 것.

## 3. 코딩 규칙 — 암묵 fallback 금지 (rule 6/7 가장 중요)

6. **암묵적 fallback/default 금지**. 특히 아래 패턴을 경계한다:
   - `.get(key, non_None_default)` — 0, 0.0, "", [], {} 같은 non-None default
   - `or {}`, `or []`, `or 0`, `or 0.0`, `or ""`
   - silent `continue` in loop (로그 없이 skip)
   - silent `return None` / `return {}` / `return []`
   - 광범위 `except (KeyError, TypeError): return None`

7. **필수 설정값이 없으면 조용히 넘어가지 말고 즉시 명확한 에러**(`KeyError`/`ValueError`)를 발생시킨다.

8. 함수/클래스 단위로 모듈화한다.

9. **`1 파일 1 기능(단일 책임 원칙)`** 을 최대한 지킨다. 단일 파일 500줄 초과 금지가 기본 목표.

10. **실험/분석용 코드가 운영 경로를 오염시키지 않도록 경계 분리**한다.
    운영 함수 (orchestrator) 에 실험/분석 inline 블록을 추가하지 않는다.
    별도 패키지로 분리하고 훅으로 호출한다.

11. 하드코딩된 stale 값, 임시 상수, 과거 결과 재사용으로 비교 산출물을 만들지 않는다.
    비교 산출물은 항상 같은 실행에서 새로 생성한다.

### rule 6/7 판정 4-카테고리 (fallback 분류 기준)

`.get(k, default)` 를 발견할 때마다 아래 4개 중 하나로 **명시 분류**한다:

| 카테고리 | 정책 | 주석 필수 여부 |
|---|---|---|
| **REQUIRED** | 필수 필드 — 직접 subscript + `KeyError` raise | 불필요 |
| **OPTIONAL** | None이 legitimate — explicit `if x is None` 분기, `or {}` 금지 | 권장 |
| **WHITELIST (display)** | 사용자 표시용 `'N/A'`/`'-'` — markdown/UI 렌더링 | **필수** (인라인 또는 모듈 docstring) |
| **WHITELIST (math)** | 수학적 정의 (미보유 → value=0 등) | **필수** (인라인) |
| **ACCUMULATOR** | 초기화 (`total = 0`, `items: List = []`) | 불필요 |

## 4. 코드 수정 후 필수 검증

12. 코드 수정 후 반드시 아래를 실행하고 결과를 보고한다.
    - `black --check` (또는 `black` 으로 자동 포맷)
    - `flake8`
    - 필요 시 `py_compile`
    - 산출물 재생성이 필요한 경우 **반드시 해당 테스트/백테스트를 재실행**하여 결과 확인

13. **"기능이 돌아간다" 만으로 통과시키지 말고, 구조 품질도 같이 검증한다.**

## 5. 검증 보고 형식 (엄수)

모든 작업 완료 보고는 **반드시 아래 2개 섹션**으로 나눠서 작성한다. 하나만 쓰지 말 것.

### A. 기능/산출물 검증

- 지시문 요구사항 충족 여부
- 최신 산출물 정합성
- 수치/메타/UI 일치 여부
- 금지사항 위반 여부
- commit/push 결과

### B. 구현 규칙/구조 품질 검증

- 암묵 fallback 금지 준수 여부 (rule 6/7)
- 단일 책임 원칙 위반 여부 (rule 8/9)
- 파일 책임 과다 여부 (500줄 기준)
- 함수/클래스 모듈화 수준
- 실험 코드와 운영 코드의 경계 분리 여부 (rule 10)
- 배포/실행 경로 영향 여부
- 기술부채나 구조적 퇴행 여부

## 6. 단계 완료 시 commit + handoff 자동 제안

14. 각 STEP/Phase 완료 보고 **직전**에 자동으로 아래를 제안한다 (명시 지시 없어도):
    - `docs/handoff/` 에 해당 챕터의 종료 + 다음 챕터 진입 문서 작성 여부
    - `git commit` 수행 여부 (push는 별도 승인)

15. 대규모 작업(여러 STEP) 진행 중에는 **매 STEP 종료 시 사용자 승인 후 다음 STEP 진입**.
    묶어서 한 번에 처리하지 말 것.

## 7. 정직한 보고 (핵심 원칙)

16. **B 섹션에서 위반 발견 시 "완료"/"통과" 로 위장하지 않는다.**
    "기능 통과 / 구조 미흡" 으로 정직하게 보고한다.

17. "전수 제거 완료", "완전 준수" 같은 **포괄적 선언은 실제 증거 없이 하지 않는다**.
    grep 결과, diff 결과, 파일 크기 등 구체적 수치로 증명한다.

18. 사용자가 과장을 지적하면 **변명하지 말고 즉시 재감사 + 보강**한다.

## 8. 중요한 원칙 (최종)

이 프로젝트에서는 **기능 구현 검증만으로는 충분하지 않다**.
반드시 "코드 품질 / 구조 품질 검증" 도 함께 수행해야 하며,
문제가 있으면 기능이 맞아도 통과로 처리하지 않는다.

## 이 규칙이 등장한 배경 (context)

이전 세션에서 사용자가 이 규칙을 고정한 이유:

1. **P207/P208/P209 god module 사고**: `run_backtest.py` 에 P207 sweep inline
   블록이 ~115줄, P208/P209 가 그 위에 더 누적되어 `run_cli_backtest` 함수가
   800줄을 넘는 god function 이 됨. `drawdown_contribution.py` 는 1064줄
   god module 로 17개 함수 6개 책임 혼재.

2. **암묵 fallback 사고**: `_summarize_selection_quality([])` 가 numeric 필드를
   0.0 으로 가득 채운 dict 를 반환해 "0 이벤트" 와 "데이터 없음" 을 구분 못 하게
   함. `raw_result.get("nav_history", []) or []` 이중 fallback 등.

3. **반복적 과장 보고 사고**: "전수 제거 완료" 를 주장했지만 실제로는 일부만
   수정된 상태. 사용자가 3회 (R5 → R5v2 → R5v3) 재검수해서 강제로 보강시켜야
   했음.

4. **god view 사고**: `pc_cockpit/views/workflow.py` 가 740줄로 P207+P208+P209
   UI inline 블록 누적. `parameter_editor.py` 도 동일.

5. **cleanup 누락 사고**: STEP 완료 후 6개 단계 동안 commit/handoff 갱신을
   한 번도 안 해서 새 세션에서 P206 끝난 시점으로 보이는 상태가 됨.

각 사고마다 위 규칙이 하나씩 강화되었다. 새 세션에서도 이 맥락을 이해하고
같은 실수를 반복하지 않아야 한다.
